Make Your CLI Agent-Friendly With These 10 Design Principles

Source ↗ Agents

What This Skill Is

If you ship any command-line tool — for a SaaS product, an internal tool, or an open-source library — there's a real chance an AI agent is going to call it before a human does. Most CLIs were designed for humans typing at a terminal: interactive prompts, ANSI colors, friendly error messages. All of that is friction for an agent. This skill covers 10 design principles (from Cloudflare and HeyGen's CLI design) for building CLIs that work for both humans AND agents.

Why It Works

The fastest way to find your CLI's weaknesses is to let an agent use it. Wherever the agent gets stuck or has to ask for help is a place your CLI is failing the agent test. Interactive prompts hang agents. ANSI color codes corrupt structured output. Vague error messages leave agents guessing. Each of these 10 principles directly removes one of these failure modes.

How to Use It

The table-stakes principles (must-haves):

  • No interactive prompts that block input by default
  • A --json flag for structured output
  • Error messages with structured "what failed, why, what to try next"

The advanced principles (separates good from great):

  • Consistent verbs: list, get, create, update, delete everywhere
  • --wait for async operations (no polling required)
  • Idempotent commands (agents retry — idempotency makes that safe)
  • Specific non-zero exit codes (agents branch on exit codes)

The agent test: Give an agent a multi-step task that uses your CLI and watch where it gets stuck. That's almost always a design flaw, not an agent flaw.

Prompt Template

You're auditing the [CLI name] CLI for agent-friendliness.

Walk through this 10-principle checklist for the CLI. For each principle,
check: does the CLI satisfy it? If yes, give an example command that
demonstrates it. If no, describe what would need to change.

Principles:
1. Non-interactive defaults — no prompts that block waiting for input
2. Structured JSON output — a --json flag that emits machine-parseable output
3. Actionable errors — what failed, why, and what to try next (in structured form)
4. Consistent vocabulary — list, get, create, update, delete (same verbs everywhere)
5. --wait support for async operations — block until done (with timeout)
6. Persistent profiles — auth, default region, default project set once
7. Idempotent commands — re-running shouldn't break things
8. Discoverable schemas — list all commands, flags, and output formats programmatically
9. Stable exit codes — 0 for success; specific non-zero codes for specific failures
10. Streaming-friendly — support pagination or streaming for large outputs

End with a prioritized list of the 3 highest-impact changes the CLI team
should make to improve agent compatibility.
