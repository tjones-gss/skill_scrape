PR Descriptions Are AI Training Data (If You Write Them That Way)

Source ↗ Content

What This Skill Is

Every AI coding session you close loses its reasoning forever. Two weeks later, nobody knows which prompt fixed the bug, what failed first, or why you chose that approach. This skill adds an "AI Context" block to every non-trivial PR description — turning pull requests into a training ground for your whole team.

The key insight: if you had to edit the AI's output, that's a signal worth capturing. It's feedback your team can turn into a permanent rule (e.g., added to CLAUDE.md).

Why It Works

Think of a PR description as a commit message for your AI reasoning. A commit records what changed. An AI-trace PR records how and why the AI got there. Teams that do this get smarter over time — every merged PR makes the next session start with more context instead of from zero. The "Manual corrections" field is especially powerful: it documents the gap between what the AI generated and what actually shipped.

How to Use It

  1. Add the template above to .github/PULL_REQUEST_TEMPLATE.md in your repo
  2. For every non-trivial AI-assisted PR, fill in all five fields before merging
  3. When you find a "Reusable rule for next time," immediately add it to CLAUDE.md
  4. Review the AI Context blocks in your team's quarterly retros — they're a window into what the model keeps getting wrong

Prompt Template

## AI Context

**Tool / model:** [Claude Code / Cursor / Codex / etc.]

**Key prompt that unlocked it:**
> [paste the prompt]

**What the AI tried first that didn't work:**
- [list failed approaches and why]

**Manual corrections:**
- [edits you made to the AI's output, with brief reason]

**Reusable rule for next time:**
- [one sentence your team should add to your CLAUDE.md or AGENTS.md]
